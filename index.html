Executive Summary
Abstract — Problem, Analysis & Solution
Problem

Between April 11–14, 2025, StreamWave ($8.2M MRR, 850K subscribers) experienced a 31 percentage point authorization collapse in Egypt and Saudi Arabia — from 78% to 47% overall. 8,018 transactions declined in 4 days. CS tickets up 340%. ~$1.2M/month in recurring revenue at risk.

Analysis Method

Cross-referenced 6 data tables (country, payment type, decline code, BIN, timeline, retry) with acquirer emails, deploy logs, and Yuno analytics. Consulted Confluence (9 pages), Slack #product-tech (known bug), GitHub (4 repos including subscription-ms), and Yuno API docs. Red herring (UI deploy) definitively ruled out via Table 2 BIN uniformity analysis.

Solution

Three-layer fix: (1) Add stored_credentials object to MIT API payload + fix Yuno bug on Checkout.com challenge_indicator; (2) Enable recurring_operation SCA exemption in Smart Routing per acquirer node; (3) Recover impacted subscriptions via Yuno's subscription engine — PAUSED→ACTIVE resume + force retry for active failures + re-enrollment campaign for canceled. Full recovery: ~$630K+/month.

On April 14, 2025, StreamWave's Head of Payments escalated a critical authorization rate collapse in Egypt and Saudi Arabia: the overall approval rate had dropped from 78% to 47% between Friday, April 11 and Wednesday, April 14. Customer support tickets had spiked 340%. Finance was asking daily for updates.

Root cause is unambiguous: Fawry (Egypt) and Checkout.com MENA (Saudi Arabia) began enforcing Strong Customer Authentication (SCA) for all Merchant-Initiated recurring transactions on April 10, 2025, as communicated in advance via email on March 28–29. StreamWave's Yuno integration was not updated to comply — recurring billing charges were submitted without the required 3DS authentication data or valid SCA exemption flags. Both acquirers responded with decline code 82 — Authentication Required.

The checkout UI redesign deployed April 11 did not cause the issue. Table 2 confirms first-time payments dropped only 2.7 pp (normal variance). The collapse is exclusively in recurring MIT payments, which do not touch the checkout UI at all. BIN uniformity across all 10 issuers (Table 4) eliminates any issuer-specific explanation.

Key finding: 8,018 code 82 declines are 100% recoverable once the MIT configuration is corrected. At $6.99/transaction, ~$56K stranded in 4 days, extrapolating to ~$420K/month. Subscription engine analysis (Yuno subscription-ms) confirms PAUSED subscriptions can be resumed programmatically with automatic next-billing-date recalculation. CANCELED subscriptions require a re-enrollment campaign. Total recoverable: ~$630K+/month.

Flow 0 — First Payment (CIT) — Correct Enrollment Pattern

Before any recurring MIT can work, the first payment must be a Cardholder-Initiated Transaction (CIT) that completes full 3DS authentication and stores the network_transaction_id. This is the anchor for all subsequent MIT charges. StreamWave uses stored_credentials.usage: "FIRST" on signup. Note: do not request low-value SCA exemption on this CIT — the exemption may be granted (no 3DS performed) and no network_transaction_id will exist for future MITs.

Flow 4 — Subscription Recovery (Yuno Subscription Engine)

Based on analysis of yuno-payments/subscription-ms, StreamWave's impacted subscriptions fall into three states, each requiring a different recovery action. CANCELED is a terminal state — there is no reverse transition. PAUSED subscriptions can be resumed programmatically with automatic billing date recalculation.

Subscription StateVolume (est.)Recovery PathAction
ACTIVE — last payment failed (code 82) ~5,600 subs Force retry after config fix Apply MIT fix → call force retry endpoint → immediate execution, same billing cycle
PAUSED — suspended due to payment failures ~1,800 subs Resume → ACTIVE Call resume endpoint → system recalculates next billing date → scheduler re-queued automatically
CANCELED — terminated due to hard decline or manual action ~620 subs Re-enrollment campaign required CANCELED is terminal. Create new subscription after customer re-authenticates via 3DS (CIT) to capture new network_transaction_id
Key insight from subscription-ms: Code 82 is a soft decline (retryable) — the retry engine's stopOnHardDecline flag should not have halted retries. Scheduled retries are still queued but failing for the same root cause. Once the MIT fix is deployed, pending retries will succeed on their next execution window. Force-retry via API skips the wait and recovers revenue immediately.

CANCELED is a terminal state in Yuno's subscription engine. Per subscription-ms state machine: the only allowed transitions are CREATED/ACTIVE/PAUSED → CANCELED. There is no reverse. Subscriptions canceled due to the April 10 payment failures cannot be reactivated — a new subscription object must be created for each affected customer, but only after they re-authenticate via a new CIT (3DS) to generate a fresh vaulted_token and network_transaction_id.

Mini Glossary

3DS / 3DS2
3D Secure — authentication protocol adding a verification layer (OTP, biometric) before card authorization. 3DS2 supports frictionless (no user action) and challenge flows.
SCA
Strong Customer Authentication. Regulatory requirement mandating two-factor authentication for online card payments. Implemented via 3DS2 in MENA.
MIT
Merchant-Initiated Transaction. A charge where the merchant initiates billing without the customer being present — e.g., a monthly subscription renewal. Requires stored_credentials to identify to issuers.
COF
Credential-on-File. A tokenized card stored for future recurring charges. StreamWave uses Yuno Vaults for COF storage.
Decline Code 82
"Authentication Required." Issuer is declining the transaction because 3DS authentication data is missing but required. 100% recoverable by submitting the correct 3DS data or valid SCA exemption.
ECI
Electronic Commerce Indicator. Authentication result code: 05 = frictionless with liability shift; 06 = challenge with liability shift; 07 = no auth, no liability shift.
Acquirer
The payment processor routing transactions on behalf of the merchant. StreamWave uses Fawry (Egypt), Checkout.com MENA (Saudi/UAE/Kuwait), Network International (Jordan/Morocco), Flutterwave (East Africa).
Issuer
The bank that issued the customer's card. Examples: Al Rajhi Bank, Banque Misr, National Bank of Egypt. The issuer makes the final approve/decline decision.
stored_credentials
Yuno API object identifying a payment as a subsequent MIT. Required fields: reason (SUBSCRIPTION), usage (USED), network_transaction_id (from first payment).
recurring_operation
SCA exemption type for scheduled recurring payments at fixed intervals. Configured at the Yuno routing level via three_d_secure_exemptions field.
Soft Decline
A retryable decline — the card is valid but temporarily declined (insufficient funds, issuer unavailable, etc.). Examples: codes 51, 05, 91, 61.
Hard Decline
A permanent decline — the card itself is invalid and retrying will fail again. Examples: codes 14 (invalid number), 54 (expired). Requires customer to update payment method.
